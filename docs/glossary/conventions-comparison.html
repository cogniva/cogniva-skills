Glossary Conventions — Comparison & Merge Options

DECIDED 2026-06-11: OPTION A — see docs/adr/0001 2026-06-11 cogniva marketplace

TL;DR: The two conventions disagree on where the glossary lives and what an entry looks like, but their behaviors are complementary: the cogniva-skills glossary skill governs lookup discipline (check the glossary before searching the codebase), while the repo-foundry convention governs communication (use and link glossary terms in every discussion, render them in plan HTML). Recommendation: Option A — one merged convention keeping docs/glossary/README.md as the location, adopting the CONTEXT.md content rules, and updating the skill's paths.

3. Merge options

Option A — One merged convention RECOMMENDED

Location wins from repo-foundry, content rules win from CONTEXT.md, behaviors are kept from both.

  • Canonical file stays docs/glossary/README.md (stable entry point, per-term anchors, plan-to-html compatibility, IDE-clickable links).
  • Entry format: ## Term heading + 1–2 sentence definition + adopt _Avoid_: alias lines + optional Mermaid + cross-links. Adopt "what it IS, not what it does", "no implementation details", "no general programming concepts".
  • Multi-context: per-context files docs/glossary/<context>.md; the index doubles as the context map, adopting the Relationships section from CONTEXT-MAP.md. Contexts usually correspond to Modules.
  • Update the glossary skill: activate on docs/glossary/README.md (keep CONTEXT.md for back-compat), search the docs/glossary/ tree. Keep its glossary-before-search lookup gate unchanged — it complements the linking protocol rather than conflicting with it.
  • Update policy: propose-then-confirm, applied immediately after confirmation (both sides effectively do this already).
  • Bump cogniva-skills to 0.2.0; document the convention in docs/strategy.md; record the decision as ADR 0001 via auto-doc.

Cost: one skill edit + format touch-up of existing glossaries. Risk: low — back-compat keeps CONTEXT.md repos working.

Option B — Adopt CONTEXT.md wholesale

  • Move docs/glossary/README.md content into root CONTEXT.md; skill untouched.
  • Loses: per-term anchors (bold terms aren't linkable), the stable docs/ entry point, and plan-to-html term auto-linking would need a different parser; root-level file crowds the repo root in every project.
  • Gains: zero changes to the existing skill.

Cost: rework of repo-foundry's converter/templates + the documented protocol. Risk: medium — breaks the linking workflow the team already uses.

Option C — Pointer bridge (status quo plus glue)

  • Keep both: docs/glossary/README.md stays canonical; a one-line root CONTEXT.md points to it ("The glossary lives at docs/glossary/README.md"), which also activates the skill.
  • Gains: no edits to either convention today.
  • Loses: permanent two-file indirection in every repo; the skill's instructions still talk about CONTEXT.md formats that nobody maintains; confusion persists for new team members.

Cost: trivial now, paid forever. Risk: low short-term, drifts long-term.

4. If Option A is chosen

  1. Edit plugins/cogniva-skills/skills/glossary/SKILL.md + CONTEXT-FORMAT.md: activation paths, search paths, entry format (h2 headings + _Avoid_ lines), index-as-context-map.
  2. Add _Avoid_: lines to this repo's glossary entries where aliases exist (e.g. Module — avoid: feature, slice, bounded context).
  3. Bump cogniva-skills version to 0.2.0 in plugin.json and the marketplace entry.
  4. Record as ADR 0001 in docs/adr/ (the auto-doc skill's format).
  5. Remove the "unify glossary conventions" item from the strategy roadmap (it lands in the repo-foundry plan's Task 10 strategy doc).
