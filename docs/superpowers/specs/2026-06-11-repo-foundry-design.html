1. This repo's purpose & layout

This repo is a local Claude Code plugin marketplace holding one plugin (repo-foundry) plus its own docs. Any future repo runs /plugin install repo-foundry once and gets all skills, hooks, and templates — fixes made here propagate, nothing drifts.

NewRepo/
  .claude-plugin/marketplace.json
  plugins/repo-foundry/
    .claude-plugin/plugin.json
    skills/
      repo-init/SKILL.md        # scaffold a brand-new repo
      add-module/SKILL.md       # add a Module to an existing repo
      plan-to-html/SKILL.md     # convert a plan/spec md -> HTML
    hooks/hooks.json            # auto-trigger plan-to-html
    scripts/convert-plan.ps1    # the md -> HTML converter
    templates/                  # glossary seed, CLAUDE.md, module layout, HTML/CSS
  docs/
    strategy.md                 # conventions + tooling decisions
    glossary/README.md          # seeded with the terms below

2. Architecture conventions

Baked into the templates and the generated repo CLAUDE.md:

  • Vertical slices are called Modules: src/Modules/<Name>/
  • Hosts (src/Hosts/Web, src/Hosts/Wpf) are composition roots — each registers either the in-process Application or the HTTP Client implementation of a Module's Contracts
  • UIs are always Blazor; the same Module UI runs in a WPF host (BlazorWebView) or a web host because it depends only on Contracts
  • Tests mirror modules under tests/
ProjectRoleMay reference
<Name>.ContractsPure public surface: interfaces, DTOs, integration events. The only thing other Modules and UIs reference.nothing
<Name>.DomainEntities, value objects, domain logicnothing
<Name>.ApplicationUse cases; in-process implementation of ContractsDomain, Contracts
<Name>.InfrastructurePersistence, external servicesApplication, Domain
<Name>.Client (optional)HTTP implementation of Contracts for remote deploymentsContracts
<Name>.UIBlazor Razor class libraryContracts only
HostsComposition roots; pick Application or Client per deploymenteverything
Glossary (propose-then-confirm): seed docs/glossary/README.md with Module, Contracts, Client, Application, Domain, Infrastructure, Module UI, Host, Vertical Slice — 1–2 sentences each plus the dependency diagram above. repo-init seeds the same glossary into every new repo. Design approval confirms this.

3. The three skills

SkillWhat it does
repo-init Interactive scaffold: git init, solution + src/Modules / src/Hosts / tests layout, glossary seed, repo CLAUDE.md with the dependency rules, .gitignore / .editorconfig, first commit.
add-module Scaffolds one Module's projects (Client offered but skippable), wires them into the solution, adds a glossary entry for the new Module.
plan-to-html Converts a markdown plan/spec into a single self-contained HTML file (CSS + vendored marked.js / mermaid.js inlined — no network, portable, mailable). Clean typographic layout, rendered Mermaid, collapsible h2 sections (expanded by default), and auto-linked glossary terms (first whole-word occurrence per section links to its entry).

4. Plan-to-HTML automation

  • PostToolUse hook on Write/Edit; script checks the path against docs/plans/**/*.md and docs/superpowers/specs/**/*.md
  • On match: convert-plan.ps1 regenerates the sibling .html (idempotent, overwrites)
  • Hook feeds context back so Claude reports “HTML ready” and offers to open it in your browser
  • Conversion failure never blocks the write — it warns and moves on

5. Testing & verification

  • Converter: sample plan fixture (headings, tables, Mermaid, code blocks) + Pester test asserting valid self-contained output
  • Skills: verified per the writing-skills checklist; hook tested by writing a dummy plan and confirming HTML appears
  • repo-init / add-module: dry-run against a temp directory before first real use

6. Non-goals (for now)

  • Remote marketplace / distribution — local path install only
  • Enforcing dependency rules via Roslyn analyzers or ArchUnitNET — CLAUDE.md rules first; analyzers are a roadmap note in strategy.md
  • NuGet packaging of module templates
